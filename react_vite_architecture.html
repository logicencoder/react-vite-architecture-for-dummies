2. Files & Folders — What each one does and why it exists
📦
package.json
Master dependency list + CLI command definitions
→Tells npm which libraries to download and what npm run dev actually means. Without this file, your project doesn't exist.
  • "dependencies": { react, react-dom }RUNTIME
    Libraries shipped to the browser. React itself lives here. Users download these.
  • "devDependencies": { vite, typescript }BUILD ONLY
    Tools used only during development. Never shipped to users. Vite, TS compiler, plugins.
  • "scripts": { dev, build, preview }
    Defines what "npm run dev" means → runs "vite". Without this, typing npm run dev fails.
  • "^18.2.0" — the ^ symbol
    Accept 18.2.0 or any newer minor version, but NOT 19.0.0. Allows bug fixes, blocks breaking changes.
  • Do you edit it?YES
    Every time you add a new library with npm install X, this file is automatically updated.
🔒
pnpm-lock.yaml
Exact version snapshot — auto-generated, never edit
→Freezes exact library versions so every developer installs the identical code. Never touch it — it manages itself.
  • What it stores
    While package.json says "react ^18.2.0", lock file records the EXACT installed version: react 18.2.0, scheduler 0.23.0, react-dom 18.2.0 — and their dependencies too.
  • Why it matters
    Makes your project reproducible. Your colleague runs npm install → gets byte-for-byte the exact same packages. "Works on my machine" becomes "works on all machines."
  • npm vs pnpm vs yarn
    package-lock.json = npm | pnpm-lock.yaml = pnpm | yarn.lock = yarn. Same concept, different format.
  • Do you edit it?NEVER
    Auto-managed. Always commit it to Git. Never delete it unless you want to force a full reinstall.
🔍
tsconfig.json / tsconfig.app.json / tsconfig.node.json
TypeScript rulebook — how strict should type checking be
→Tells TypeScript how strictly to check your code for errors. Three files because your React UI code and your config files run in different environments.
  • tsconfig.json (root)
    Parent file. Doesn't set rules itself — just points to the two sub-configs below.
  • tsconfig.app.jsonBROWSER
    Rules for your React UI code — App.tsx, components. Has "jsx": "react-jsx" to understand JSX syntax.
  • tsconfig.node.jsonNODE
    Rules for config files like vite.config.ts that run in Node, not the browser. Different environment = different rules.
  • "strict": true
    The most important setting. Turns on all safety checks. TypeScript will catch bugs before they reach the browser.
  • Do you edit it?RARELY
    Leave as-is for most projects. You'd touch it only to add paths, adjust strictness, or configure special imports.
⚙️
vite.config.ts
Build engine control panel — the most critical config file
→Configures Vite — the tool that compiles your code. The one line that matters most: plugins: [react()] — without it, nothing works.
  • plugins: [react()]CRITICAL
    THE most important line in this file. Without it, Vite doesn't understand JSX or .tsx files. Every <div> in your code is a syntax error. This plugin bridges React and Vite.
  • server.port: 5173
    The dev server port. Change it here if 5173 is taken. localhost:5173 = your running app.
  • build.outDir: 'dist'
    Where final built files go after "npm run build". This is what you deploy to a server or Netlify.
  • defineConfig()
    A TypeScript helper that gives you autocomplete while editing this file. Not mandatory, just helpful.
  • Do you edit it?SOMETIMES
    When adding Vite plugins (TailwindCSS, PWA, etc.) or changing server settings. Otherwise leave alone.
🚫
.gitignore
Git blocklist — files that NEVER go to GitHub
→A list of files Git completely ignores. Most important entry: node_modules/ — 300MB of code nobody should upload.
  • node_modules/NEVER COMMIT
    The most critical entry. 300MB+ of auto-generated package code. Anyone can regenerate it with "npm install" in 10 seconds. Uploading it to GitHub wastes everyone's time.
  • dist/NEVER COMMIT
    The built output from "npm run build". Regenerated automatically. No point tracking it in Git.
  • .env / .env.localSECURITY
    Environment variables — API keys, database URLs, secrets. If these go to GitHub, anyone can see them. Always in .gitignore.
  • Do you edit it?SOMETIMES
    When you add new secret files or folders you don't want tracked. Add the filename/folder name, one per line.
⚫
node_modules/
The "black hole" — actual library source code, 300MB+
→Where npm puts the real code of every library after you run npm install. Never touch it. Delete it anytime — restores itself in seconds.
  • Created byAUTO-GENERATED
    "npm install" reads package.json and fills this folder with the real source code of every library. Hundreds of sub-folders, thousands of files.
  • import React from 'react'
    When you write this in your code, Vite goes to node_modules/react/index.js to get the actual React source. The import is a shortcut. The code is in here.
  • Contains ~800 packages
    React depends on scheduler. Vite depends on esbuild, rollup, chokidar. Each of those has more dependencies. You install 5 things, you get 800.
  • Do you edit it?NEVER
    Never open it, never edit anything inside. Delete the whole folder freely — "npm install" rebuilds it in seconds.
📂
public/
Static assets — served raw, zero processing by Vite
→Drop files here that you want served exactly as-is — favicons, images, fonts. Reference with /filename in code. No import needed.
  • How it works
    Files here bypass Vite entirely. Copied to the final build as-is. Reference with a leading slash in your code: <img src="/logo.svg" />
  • What goes here
    favicon.ico, robots.txt, font files, images that never change, any static file you want available at a direct URL.
  • vs src/assets/
    Files in src/assets/ are processed by Vite (hashed, optimized). Files in public/ are not. Use public/ for things that must keep their original filename.
  • Do you edit it?YES
    Add your favicon, app icons, any static asset that needs a direct URL. No import needed — just reference with /filename.
💻
src/
Your source code — 99% of your actual work lives here
→Your workspace. Every component, every line of logic, every stylesheet you write goes in here. Vite compiles everything inside src/ before the browser ever sees it.
  • Processed by ViteCOMPILED
    Everything here gets compiled. TypeScript → JavaScript. JSX → function calls. CSS → bundled stylesheet. The browser never sees .tsx files directly.
  • src/main.tsxENTRY POINT
    The bridge. Finds the #root div in index.html and mounts React into it. This is where JavaScript takes over the page.
  • src/App.tsxROOT COMPONENT
    Top of the component tree. Everything you build branches from here. The counter in the video lives here.
  • src/components/
    You create this folder yourself. One file per component: Button.tsx, Header.tsx, Sidebar.tsx. Import them into App.tsx.
  • Do you edit it?ALWAYS
    This is your workspace. Every component, every hook, every stylesheet you write goes inside src/.
3. Injection Chain — What happens step by step from blank HTML to running app
Step 1 🌐 Browser loads index.html from Vite dev server
Vite serves this file when you open localhost:5173. The page is a shell — almost completely empty.
Step 2 📄 index.html — the skeleton with one empty div
<div id="root"></div> — this is the ONLY thing that matters in this file. Empty. React will fill it. Also loads main.tsx via <script> tag.
Step 3 🌑 At this moment — the page is completely blank
HTML loaded, div exists, but React hasn't run yet. If you disable JavaScript, you'd see an empty white page forever. This is normal.
Step 4 🔗 main.tsx runs — the bridge connects React to the HTML
ReactDOM.createRoot(document.getElementById('root')!) — finds that div. Hands control of it to React. React now owns everything inside #root.
Step 5 ❗ The ! after getElementById — TypeScript assertion
getElementById() can return null if the element doesn't exist. The ! tells TypeScript "I guarantee it exists, stop warning me." Without it, TypeScript error.
Step 6 ⚛️ .render(<App />) — React renders your component tree into #root
React converts App.tsx and all its children into real DOM elements and inserts them into the empty div. The page is now alive.
Step 7 🖱️ User clicks button → useState triggers re-render
setCount(count + 1) → React compares old vs new component tree → only updates the changed DOM node. No page reload. No full re-render. Just the diff.
Step 8 ✅ DevTools → Elements — proof that React injected everything
<div id="root"> is now full of your components' HTML output. It was empty before main.tsx ran. Everything inside it is React-managed.
4. Key Concepts — Terms you need to know
JSX
What it looks like: <div>Hello</div>
What it actually is: JavaScript syntax
Who processes it: Vite + react() plugin
Compiled to: React.createElement()
Not HTML. Not a template. Pure JavaScript disguised as HTML.
useState
What it is: React Hook
Usage: const [count, setCount] = useState(0)
count = current value (read only)
setCount = function to change it
Every setCount() call triggers a re-render of this component.
Component
What it is: A JavaScript function
Returns: JSX (looks like HTML)
Example: function App() { return <div>...</div> }
Used as: <App /> anywhere in JSX
Capital letter = component. Lowercase = real HTML tag.
Hot Reload (HMR)
What it is: Vite feature
What it does: Updates browser on file save
Without reload: Preserves state
How fast: ~50ms update
Save a .tsx file → browser updates instantly. No F5 needed.
StrictMode
Where: main.tsx wraps <App />
In development: renders twice to find bugs
In production: completely removed
Purpose: catch side effects early
If your effect runs twice in dev, StrictMode is why. Normal behavior.
Props
What they are: Parameters for components
Passed as: <Button color="blue" />
Received as: function Button({ color })
Direction: parent → child only
Data flows DOWN the tree. Child cannot modify parent's state via props.

6. Things you must understand — Key mental models
⚛️ The Injection Chain
React doesn't magically appear. It follows a fixed 3-step chain every time.
WRONG mental model
"React generates the HTML page"
CORRECT mental model
index.html → main.tsx → App.tsx → fills #root div
📦 node_modules is not your code
It's a cache of library code. Treat it like a generated build artifact.
WRONG mental model
Editing files in node_modules
CORRECT mental model
Delete it. npm install. Back in seconds.
🔄 JSX is NOT HTML
JSX looks like HTML but it's JavaScript. Vite compiles it before the browser ever sees it.
You write (JSX)
<button onClick={fn}>Click</button>
Browser gets (JS)
React.createElement('button', {onClick: fn}, 'Click')
💾 State vs Variables
Regular variables reset on every render. useState persists between renders.
BROKEN — resets every render
let count = 0; count++
CORRECT — React remembers it
const [count, setCount] = useState(0)
🔧 public/ vs src/assets/
Two places for files — completely different treatment by Vite.
public/ — raw, no processing
<img src="/logo.svg" /> ← direct URL
src/assets/ — processed by Vite
import logo from './assets/logo.svg'

7. How it all works — The big picture in plain words
The whole thing in one paragraph

You write code inside src/. Vite — configured by vite.config.ts — reads your TypeScript and JSX, compiles it into plain JavaScript the browser understands, and serves it live at localhost:5173. When the browser opens that page, it receives index.html — an almost empty shell with one div. That div triggers main.tsx, which mounts your entire React app into it. Every component, every button, every state change — React manages it all inside that one div, updating only what changed, without ever reloading the page. The config files set the rules. node_modules/ holds the tools. public/ holds static files. Everything else is src/.

⚡ What starts what
The sequence when you type npm run dev
1 npm reads package.json → finds the "dev" script
2 Runs vite → reads vite.config.ts
3 Vite compiles src/, uses tsconfig for TS rules
4 Browser gets index.html → runs main.tsx
5 App.tsx mounts → your UI appears in browser
